紧密耦合代码是指组件之间过度依赖彼此的实现细节。
这种依赖会使你的代码更难维护，灵活性更低，并且更容易出现错误。


一个常见的解决方案是使用接口来解耦组件并提高模块化。

## 紧密耦合代码

紧密耦合代码指的是一个组件严重依赖于另一个组件的内部细节的情况。这会创建一个刚性系统，在不破坏现有功能的情况下很难修改或扩展。

## 直接依赖示例

考虑一个与`Library`类直接交互的`BorrowableBook`类。`Library`类依赖于`BorrowableBook`的具体实现来执行借用操作。

```c#
public class BorrowableBook
{
    public string Title { get; set; }
    public bool IsAvailable { get; private set; } = true;

    public BorrowableBook(string title)
    {
        Title = title;
    }

    public void Borrow()
    {
        if (IsAvailable)
        {
            IsAvailable = false;
            Console.WriteLine($"You have borrowed \"{Title}\".");
        }
        else
        {
            Console.WriteLine($"\"{Title}\" is already borrowed.");
        }
    }
}

public class Library
{
    private BorrowableBook _book;

    public Library(BorrowableBook book)
    {
        _book = book;
    }

    public void BorrowBook()
    {
        if (_book.IsAvailable)
        {
            _book.Borrow();
        }
        else
        {
            Console.WriteLine("The book is not available.");
        }
    }
}
```

在这段代码中，`Library`类直接依赖于`BorrowableBook`类。

- `直接依赖`：`Library`类与`BorrowableBook`类型紧密耦合。  
如果你想添加新的可借阅项目（例如`DVD`、`杂志`），你需要修改`Library`类。
- `有限的扩展性`：当前的设计使得在不修改现有代码的情况下扩展功能变得困难。  
这种紧密耦合违反了`开闭原则（OCP）`，该原则指出软件应该对扩展开放但对修改封闭。  
换句话说，你应该能够在不修改现有代码的情况下添加新功能。

通过减少依赖，你可以：
- `提高可维护性`：系统一部分的变更不太可能影响其他部分。
- `增强灵活性`：可以添加新功能而不修改现有代码。
- `简化测试`：组件可以独立测试。
